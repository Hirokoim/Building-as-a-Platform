# Building as a Platform (BaaP)

「アプリを配る側」から「アプリが作られる前提条件（文脈・データ・ルール）を提供する側」へと転換する構想。
Vibe Crafting 時代における不動産デベロッパーのあるべきポジションを定義する。

**Status:** 実装フェーズ開始。テナント側AIは MulmoClaude を採用し、Context Layer 用の MCP サーバーの試作を開始。
現在動いているのはダミーデータ版の MCP サーバーで、次のステップとして Supabase への差し替えを予定している（詳細は「現在実装しているアーキテクチャ」参照）。

> **このREADMEの読み方**
> 本書には2つの異なる時間軸のアーキテクチャが登場する。
> - 「コンセプト／アーキテクチャ（将来ビジョン）」＝ 4〜7年スパンで目指す構想上の完成形
> - 「現在実装しているアーキテクチャ」＝ 今まさにコードを書いて動かしている、Supabaseを使った現実的な実装
>
> 両者は矛盾しない。今の実装は、将来ビジョンに向かう最初の一歩として設計している。

## Concept

現在の不動産 SaaS は、ビル側が画面（アプリ）を作ってテナントに配る構造になっている。

```
[ビル側のSaaS] → 画面 → テナントが使う
```

本構想が描くのは、その逆転である。

```
[ビル側のデータ / API / ルール]
        ↓
[テナント側のAI]
        ↓
[テナントごとに最適化されたアプリ]
```

本質的なシフトは **「画面の所有権」がビルからテナントへ移る** こと。
ビルはアプリを作らず、アプリが作られるための材料（文脈）を提供する存在になる。

※このセクションは長期的な構想（Concept）であり、次章のアーキテクチャ図とあわせて「将来ビジョン」として読んでほしい。実際に今動いているものは、その後の「現在実装しているアーキテクチャ」を参照。

## コアコンセプト

- **三層の役割分担** — ビル＝Context Provider、テナント＝Experience Designer、AI＝Translator/Builder
- **ビルは画面ではなく文脈を提供する** — 申請ルール・館内情報・設備予約・FAQ・イベント情報を、アプリではなく API / 知識ベースとして公開する
- **アプリは「配布」されず「召喚」される** — テナント側の AI 基盤上で都度生成される
- **汎用と固有の分離** — アプリは汎用、データ・ルール・文脈はビル固有
- **Twilio / Stripe モデル** — 自らアプリを作らず、世界中のアプリから呼び出される側に回る

## アーキテクチャ（将来ビジョン）

Building as a Platform の三層構造。**これはロードマップの「長期（4〜7年）」に対応する構想上の完成形であり、今のコードベースがこの形になっているわけではない。**

```
┌──────────────────────────────┐
│ テナント側のAI               │
└──────────────────────────────┘
              ↑ API / MCP
┌──────────────────────────────┐
│ ビル側の Context Layer       │
│  - 館内ルール                │
│  - 申請仕様                  │
│  - 設備情報                  │
│  - 入居者情報（権限制御）    │
│  - イベント・周知            │
│  - 物件横断データ            │
└──────────────────────────────┘
              ↑
┌──────────────────────────────┐
│ 物理ビル（IoT / BMS / 設備） │
└──────────────────────────────┘
```

| レイヤー | 役割 |
|---|---|
| テナント側 AI | Experience Designer（体験を設計・生成） |
| Context Layer | Context Provider（文脈を提供） |
| 物理ビル | データソース（IoT / BMS / 設備） |

## 主要論点（検証中の仮説）

| 論点 | 結論 |
|---|---|
| トークン消費は誰が負担するか | 短期は議論あり。中長期でテナントが自前の法人AI契約を持ち、議論は消失するか |
| アプリ配布の問題 | 「アプリを配る」から「APIキー＋仕様を配る」へ移行する |
| ビル側がアプリを作る必要性 | 作る必要はない。データ・ルール・APIガバナンスを作るべき |
| ビル専用にする必要性 | アプリは汎用でよい。データ・ルール・文脈のみビル固有とする |

## ロードマップ

**短期（1〜2年）** ← 今ここ。下記「現在実装しているアーキテクチャ」がこのフェーズの実装物
業務標準化のため既存 SaaS を導入しつつ、データ取り出し性と API 設計を契約条件に組み込む。並行して館内申請やルールを構造化ナレッジとして整備する。

**中期（2〜4年）**
自社の「Building Context API」を保有。MCP サーバー化し、ビル横断でルール・データを統一フォーマット化。テナント向けに「ビル接続用キット」を配布する。

**長期（4〜7年）**
ビル＝Platform、テナント＝Vibe Crafter という構造へ。ビル側はデータ・空間・信頼を提供する存在となり、運営の本質が「画面提供」から「文脈提供」へ変わる。上記「アーキテクチャ（将来ビジョン）」の三層構造はこのフェーズの姿を指す。

## 現在実装しているアーキテクチャ（Supabase版）

長期ビジョンの三層構造をいきなり作るのではなく、まずは動くものを最短距離で作るための現実解。
「データを読み出す側（MCPサーバー）」と「データを書き込む側（管理用Webアプリ）」を分け、Supabaseを共有のデータベースとして両者から独立してつなぐ構成にしている。

```
┌──────────────────────────────┐         ┌──────────────────────────────┐
│ テナント側のAI (MulmoClaude) │         │ 管理用Webアプリ (今後作成)   │
└──────────────────────────────┘         └──────────────────────────────┘
              ↑ MCP (HTTP)                              │
              │ 読み出し                                 │ 書き込み
┌──────────────────────────────┐                        │
│ MCP サーバー (mcp-server/)   │                        │
│  - getBuildings              │                        │
│  - getBuildingDetail         │                        │
│  - getRequests               │                        │
│  - createRequest             │                        │
│  - データ層は store.ts に分離 │                        │
└──────────────────────────────┘                        │
              ↑ (今後差し替え)                            │
              └───────────────┬──────────────────────────┘
                              ↓
              ┌──────────────────────────────┐
              │ Supabase (PostgreSQL)        │
              │  - テーブル定義がそのままDB   │
              │  - REST / GraphQL API 自動生成│
              │  - RLSで権限制御（入居者情報等）│
              └──────────────────────────────┘
```

**MCPサーバーと管理用Webアプリを直接つながない理由**
- Supabaseを「共有のデータベース」として使うことで、両者は疎結合（お互いを意識せず独立して動く）になる
- 片方を作り直したり仕様変更しても、もう片方への影響が出にくい
- 非エンジニアが2つのアプリを並行して育てていく上でも、依存関係がシンプルな方が管理しやすい
- 例外的に直接つなぐ検討の余地があるのは「書き込み時のビジネスロジックが複雑でDB(RLS)だけでは表現しきれない場合」や「リアルタイム通知が必要な場合」だが、現時点ではどちらも該当しないため2本立て構成のままとする

**なぜSupabaseなのか（3つの理由）**
1. 実体はPostgreSQL（標準的なDB）なので、将来「独自Webアプリ＋独自DB」に移行したくなってもデータはそのまま持っていける。ロックインの心配はほぼない。
2. テーブルを作るだけでREST APIとGraphQL APIが自動生成されるため、「REST APIを自作する」工数がほぼゼロになる。MCPサーバーづくりに集中できる。
3. RLS（Row Level Security）が標準機能としてあり、将来ビジョンにある「入居者情報（権限制御）」のような要件にも最初から対応できる。

**現在の実装状況**
- `mcp-server/` に4つのMCPツール（`getBuildings` / `getBuildingDetail` / `getRequests` / `createRequest`）を実装済み。
- データ層（`store.ts`）は実際のSupabaseプロジェクトに接続済み。`buildings` / `facilities` / `requests` テーブル（`mcp-server/supabase/migrations/0001_init.sql`）に対して、curlでのハンドシェイク〜tools/call（一覧取得・詳細取得・申請作成）まで一通り動作確認済み。
- REST API層は別途作らない方針。REST を叩く他クライアントが存在しない限り、MCPツールが `store.ts` を直接呼ぶ方が単純で工数も少ない。
- MCPサーバーはHTTP（`type: "http"`）で公開。MulmoClaude側のDockerサンドボックス使用時にstdio型MCPが使えない制約があるため。

**次のステップ**
1. `~/mulmoclaude` 側で `building-context` サーバーが有効か確認し、チャットから動作確認
2. データ入力用の簡易管理画面を作成（Supabaseへの書き込み担当）
3. `tenants` テーブルを追加し、入居者情報の権限制御（RLS）を設計

> このSupabase版は、将来ビジョンでいう「Context Layer」の最小実装にあたる。三層構造をフルに実現するものではなく、まずは「MCPサーバーが1つのSupabaseプロジェクトのデータを返す」という一番小さい形からスタートしている。

## フォルダ構成

```
Building-as-a-Platform/
├── README.md            本ドキュメント（構想・アーキテクチャ）
├── Mulmo_advice.md       MulmoClaude 連携の実装メモ（設計時の下書き）
└── mcp-server/           Context Layer の MCP サーバー（テナント側AI ↔ ビル側データの橋渡し）
    ├── src/
    │   ├── store.ts      データ層。今はダミーデータだが、将来は Supabase 等への差し替え窓口になる
    │   ├── tools.ts       MCP ツール定義（getBuildings / getBuildingDetail / getRequests / createRequest）
    │   └── index.ts       HTTP(Streamable HTTP) で MCP を公開するエントリポイント
    └── package.json
```

**設計方針:**
- REST API 層は作らず、MCP ツールがデータ層（`store.ts`）を直接呼ぶ。REST を叩く他クライアントが存在しない限りこの方が単純。
- `store.ts` の関数シグネチャ（`listBuildings` / `getBuilding` / `listRequests` / `createRequest`）は固定し、中身だけを差し替えられるようにしてある。Context Layer のデータベースを Supabase にする場合も、この層をSupabaseクエリに置き換えるだけで済む。
- MCP サーバーは HTTP（`type: "http"`）で公開している。MulmoClaude 側の Docker サンドボックス使用時に stdio 型 MCP が使えない制約があるため。
