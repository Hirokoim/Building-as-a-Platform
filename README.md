# Building as a Platform (BaaP)

「アプリを配る側」から「アプリが作られる前提条件（文脈・データ・ルール）を提供する側」へと転換する構想。
Vibe Crafting 時代における不動産デベロッパーのあるべきポジションを定義する。

**Status:** 構想フェーズ。仮説と設計思想を整理中。実装コードはまだ存在しない。

## Concept

現在の不動産 SaaS は、ビル側が画面（アプリ）を作ってテナントに配る構造になっている。

```
[ビル側のSaaS] → 画面 → テナントが使う
```

本構想が描くのは、その逆転である。

```
[ビル側のデータ / API / ルール]
        ↓
[テナント側のAI]
        ↓
[テナントごとに最適化されたアプリ]
```

本質的なシフトは **「画面の所有権」がビルからテナントへ移る** こと。
ビルはアプリを作らず、アプリが作られるための材料（文脈）を提供する存在になる。


## コアコンセプト

- **三層の役割分担** — ビル＝Context Provider、テナント＝Experience Designer、AI＝Translator/Builder
- **ビルは画面ではなく文脈を提供する** — 申請ルール・館内情報・設備予約・FAQ・イベント情報を、アプリではなく API / 知識ベースとして公開する
- **アプリは「配布」されず「召喚」される** — テナント側の AI 基盤上で都度生成される
- **汎用と固有の分離** — アプリは汎用、データ・ルール・文脈はビル固有
- **Twilio / Stripe モデル** — 自らアプリを作らず、世界中のアプリから呼び出される側に回る

## アーキテクチャ

Building as a Platform の三層構造。

```
┌──────────────────────────────┐
│ テナント側のAI               │
└──────────────────────────────┘
              ↑ API / MCP
┌──────────────────────────────┐
│ ビル側の Context Layer       │
│  - 館内ルール                │
│  - 申請仕様                  │
│  - 設備情報                  │
│  - 入居者情報（権限制御）    │
│  - イベント・周知            │
│  - 物件横断データ            │
└──────────────────────────────┘
              ↑
┌──────────────────────────────┐
│ 物理ビル（IoT / BMS / 設備） │
└──────────────────────────────┘
```

| レイヤー | 役割 |
|---|---|
| テナント側 AI | Experience Designer（体験を設計・生成） |
| Context Layer | Context Provider（文脈を提供） |
| 物理ビル | データソース（IoT / BMS / 設備） |

## 主要論点（検証中の仮説）

| 論点 | 結論 |
|---|---|
| トークン消費は誰が負担するか | 短期は議論あり。中長期でテナントが自前の法人AI契約を持ち、議論は消失するか |
| アプリ配布の問題 | 「アプリを配る」から「APIキー＋仕様を配る」へ移行する |
| ビル側がアプリを作る必要性 | 作る必要はない。データ・ルール・APIガバナンスを作るべき |
| ビル専用にする必要性 | アプリは汎用でよい。データ・ルール・文脈のみビル固有とする |

## ロードマップ

**短期（1〜2年）**
業務標準化のため既存 SaaS を導入しつつ、データ取り出し性と API 設計を契約条件に組み込む。並行して館内申請やルールを構造化ナレッジとして整備する。

**中期（2〜4年）**
自社の「Building Context API」を保有。MCP サーバー化し、ビル横断でルール・データを統一フォーマット化。テナント向けに「ビル接続用キット」を配布する。

**長期（4〜7年）**
ビル＝Platform、テナント＝Vibe Crafter という構造へ。ビル側はデータ・空間・信頼を提供する存在となり、運営の本質が「画面提供」から「文脈提供」へ変わる。
